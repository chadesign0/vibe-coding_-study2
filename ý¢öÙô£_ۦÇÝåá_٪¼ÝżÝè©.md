## 버그 / 기능 오동작

### BUG-1 — 파일 업로드 입력 요소 누락 `심각`

- **위치:** `app.js:912` / `index.html` 전체
- **내용:** `app.js`에서 `document.getElementById("keywordsFile")`로 파일 입력을 참조하지만, `index.html`에 `id="keywordsFile"`인 `<input type="file">` 요소가 존재하지 않는다. 결과적으로 파일 업로드(xlsx/txt) 기능이 완전히 동작하지 않는다.
- **수정:** `index.html` 업로드 패널 안에 다음 요소 추가
  ```html
  <input type="file" id="keywordsFile" accept=".xlsx,.txt" class="input">
  ```

---

### BUG-2 — 키워드 삭제 시 런타임 config 파일 미반영 `심각`

- **위치:** `server.py:468-483` `delete_keyword()`
- **내용:** 키워드 삭제 시 기본 `CONFIG_PATH`(april_keywords.json)에서만 삭제하고, 병원별 런타임 설정 파일(`config/runtime_*.json`)에서는 삭제하지 않는다. 이후 **[전체 재채점]** 실행 시 런타임 파일에 남아 있는 삭제된 키워드가 다시 채점되어 부활한다.
- **수정:** `delete_keyword()` 내에서 `runtime_config_path_for_hospital()`로 경로를 구하고, 해당 런타임 파일에서도 keyword를 제거해야 한다.

### BUG-3 — 병원 정규화 로직 불일치 (server.py ↔ app.js) `보통`

- **위치:** `server.py:27-29` `HOSPITAL_CANONICAL` / `app.js:250-257` `canonicalHospitalName()`
- **내용:** `app.js`에는 `"삼성본정형외과"` → `"삼성본병원"` 매핑이 있지만 `server.py`의 `HOSPITAL_CANONICAL`에는 해당 항목이 없다. 프론트에서 `"삼성본정형외과"`로 채점 요청 시 서버는 canonical 변환 없이 그대로 처리해 데이터 슬롯이 어긋날 수 있다.
- **수정:** `server.py` `HOSPITAL_CANONICAL`에 추가
  ```python
  "삼성본정형외과": "삼성본병원"
  ```

### BUG-4 — 파일 핸들 미닫힘 (리소스 누수) `보통`

- **위치:** `build_april_month.py:882, 998, 1028, 1033, 1047`
- **내용:** `json.load(open(path, ...))` 및 `json.dump(..., open(path, "w", ...))` 패턴이 6곳 이상 사용되고 있으나 `with` 문을 사용하지 않아 파일 핸들이 명시적으로 닫히지 않는다. 채점 중 오류 발생 시 파일이 잠길 수 있다.
- **수정:** 모든 `open()` 호출을 `with` 구문으로 교체
  ```python
  with open(path, encoding="utf-8") as f:
      data = json.load(f)
  ```

### BUG-5 — escapeHtml에서 단일 따옴표(`'`) 이스케이프 누락 `보통`

- **위치:** `app.js:583-589` `escapeHtml()`
- **내용:** `&`, `<`, `>`, `"` 네 가지만 이스케이프하고 `'`(단일 따옴표)는 처리하지 않는다. `scoreReasonText()` 결과가 `title` 속성에 삽입될 때 키워드나 병원명에 단일 따옴표가 있으면 속성이 조기 종료되거나 레이아웃이 깨질 수 있다.
- **수정:**
  ```js
  .replace(/'/g, "&#39;")
  ```
  추가

### BUG-6 — 채점 완료 토스트 메시지 오류 `경미`

- **위치:** `app.js:954-955` `runKeywordUpload()`
- **내용:** 채점 성공 후 토스트가 `res.count`(총 병합 키워드 수)를 보여주는데, 사용자가 방금 입력한 키워드 수가 아니라 누적 전체 키워드 수를 표시한다. 50개 중 3개만 새로 추가된 경우도 `"3개 키워드 반영 완료"`로 표시되어 혼란을 준다.
- **수정:** 서버 응답에 `newCount`(신규 추가 수) 필드를 별도로 내려주거나, 메시지를 `"총 N개 키워드 반영 완료"`로 명확히 변경한다.

## 하드코딩 / 유지보수성 문제

### HC-1 — 월 탭이 1~4월로 고정 `심각`

- **위치:** `app.js:80` `FIXED_MONTH_TABS` / `app.js:297-302` `monthTabIndexForToday()`
- **내용:** 5월 이후 배점표를 지원하지 못한다. `monthTabIndexForToday()`에서 5월부터는 무조건 4월 탭(인덱스 3)을 선택하도록 `Math.min` 처리되어 있어 5월 데이터가 있어도 탭이 없다.
- **수정:** `FIXED_MONTH_TABS`를 JSON 데이터에서 동적으로 생성하거나 12개월로 확장한다.
  ```js
  const FIXED_MONTH_TABS = ["1월","2월","3월","4월","5월","6월","7월","8월","9월","10월","11월","12월"];
  ```

- **수정:** 배포 시에는 허용 출처를 명시하거나 인증 미들웨어를 추가한다.

### SEC-2 — 채점 로그 전체를 클라이언트에 노출 `낮음`

### QA-6 — 재채점 시 삭제된 키워드 잔류 `보통`

- **위치:** `build_april_month.py:1065-1067`
- **내용:** 재채점 시 기존 데이터에서 `reused_rows`(재사용 행)를 만들고 현재 config에 있는 키워드만 `fresh_keywords`로 채점한다. 그러나 이전에 삭제된 키워드가 `reused_rows`에 남아 있으면 `build_month_payload`에서 그대로 출력 rows에 포함된다. `delete_keyword_from_scoring_data()`로 scoring-data에서 삭제해도 재채점 흐름에서 다시 살아날 수 있다.
- **수정:** `build_month_payload` 호출 전 `reused_rows`를 현재 키워드 기준으로 필터링한다.
  ```python
  reused_rows = {k: v for k, v in reused_rows.items() if k in set(all_keywords)}
  ```
